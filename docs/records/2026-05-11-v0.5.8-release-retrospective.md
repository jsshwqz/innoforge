## 做得好的

1. **冲突解决效率高** — 6 个冲突文件全部一次性解决，编译通过，109 项测试全过
2. **验证流程严谨** — fmt → clippy → test → build 顺序执行，未跳过任何步骤
3. **清理彻底** — 临时文件、旧 git stash 全部清理完毕
4. **Release 说明完整** — 中英双语，结构清晰，覆盖所有变更点

## 可改进的

1. **发布流程应先确认** — 之前的 session 以为推送了 tag 就是发布成功，实际 PR 有冲突未合并到 main。下次应先检查 PR 状态（`gh pr view --json mergeable`）再做发布声明
2. **跨平台构建策略** — 目前只有 Windows 能编译，缺少自动化构建流程（CI）。后续可考虑 GitHub Actions 自动构建全平台安装包
3. **Worktree 分支管理** — 合并后应及时清理已合并的 worktree 分支和本地目录，避免残留
4. **文档不应合并为一个** — 修改记录、复盘、错误记录应分开为三个独立文档，方便查阅

## 关键教训

- **Tag 在分支上 ≠ 发布完成**：标签必须指向 `main` 分支的合并提交才是正式发布
- **PR 合并前必须检查冲突**：`gh pr view --json mergeable` 可提前发现冲突
- **Release 资产会过期**：发布后发现需要补充变更时，必须重新构建并上传安装包
- **GitHub TLS 不稳定**：偶发 TLS 连接错误，可重试或切换到 Gitee
