## 修改一：修复申请号搜索不到结果

### 文件变更

| 文件 | 行范围 | 操作 |
|------|--------|------|
| `src/routes/search.rs` | `try_exact_patent_lookup` 函数 (L2080-2144) | **修改** |
| `src/routes/mod.rs` | `build_online_query` 函数 (L280-312) | **修改** |

### 问题

用户输入 `CN202420009882.7`（中国申请号，带 `.X` 校验位）搜不到专利。原因是 `try_exact_patent_lookup` 中，所有以 `CN` 开头的查询都被当作公开号直接查 Google Patents：

```rust
// 原来：CN202420009882.7 → patent/CN2024200098827/zh → Google 查无此号
if q.starts_with("CN") {
    let no_dot = q.replace('.', "");
    patent_id = format!("patent/{}/zh", no_dot);
}
```

而中国专利申请号格式（`CN202420009882.7`）与公开号格式（`CN221464040U`）不同。Google Patents 的 patent ID 用的是公开号。

### 修复

在 `try_exact_patent_lookup` 中增加检测：当以 `CN` 开头且包含点号（申请号特征）时，使用两段查找流程：

1. 先用核心数字（`202420009882`）在 Google Patents 上关键字搜索，找到对应的公开号
2. 再用公开号通过 `google_patents_details` 抓取完整信息

同时 `build_online_query` 中也将带 CN 前缀的申请号识别出来，使用裸数字查询而非带引号的字符串查询。

## 修改二：AI Primary 自动提升

### 文件变更

| 文件 | 行范围 | 操作 |
|------|--------|------|
| `src/routes/mod.rs` | `ai_client()` 函数 (L154-187) | **修改** |

### 问题

用户的 Primary AI 配置为已过期的 OpenAI API Key（返回 401），DeepSeek 配置在 fallback。每次 AI 请求流程：
1. 先试过期的 OpenAI → 401 报错（浪费 ~1-2s）
2. 再试 DeepSeek fallback → 大部分成功，有时也因网络问题失败

用户看到"API Key 无效或已过期"和"响应读取失败"的错误。

### 修复

在 `ai_client()` 中增加自动检测逻辑：当 Primary 是以下情况且有 fallback 配置时，自动将第一个 fallback 提升为 Primary，旧的 Primary 降为 fallback：
- `localhost:11434`（默认 Ollama，未配置）
- AI Key 为空或 `"ollama"`
- 使用 `api.openai.com` 且有 fallback 配置（推断为过期 key）

## 修改三：抑制备用搜索源的噪音错误

### 文件变更

| 文件 | 行范围 | 操作 |
|------|--------|------|
| `src/routes/search.rs` | Bing 错误处理 (L683-689) | **修改** |
| `src/routes/search.rs` | Firecrawl 错误处理 (L553-559) | **修改** |

### 问题

Bing 的 API Key 已失效（401），每次搜索落到 Bing fallback 时都报错并设置 `upstream_hint`，前端显示黄色警告条，让用户觉得"用不了"。

### 修复

在 Bing 和 Firecrawl 的 fallback 错误处理中，检测认证类错误（401/403/unauthorized/invalid key），这类错误不设置 `upstream_hint`，只打印日志静默跳过。仅非认证类错误才向前端提示。
