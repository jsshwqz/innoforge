# 复盘 / Retrospective

> 日期：2026-05-11 | 版本：v0.5.8（待发） | 时长：约 2 小时（跨两轮会话）

---

## 做得好的

1. **精准执行用户需求**：用户要求"只保留 SerpAPI + DeepSeek"，所有代码改动严格按照此边界执行，没有 overscope。

2. **分阶段推进**：阶段一（搜索/AI 精简）和阶段二（体验改进）分开执行，每个阶段都有编译验证 + 运行测试，避免大范围改动一次到位导致难以定位问题。

3. **Comprehensive grep 验证**：在删除字段后，对全代码库做 grep 确认零残留引用，确保没有遗漏的编译错误。

4. **以官方文档为依据修正权限问题**：发现 `allowMode` 配置无效后，不靠猜测，直接查 Claude Code 官方设置文档获取正确格式。

5. **构建部署自动化**：使用独立的 CARGO_TARGET_DIR 避免与主仓库构建目录冲突，编译时间从首次 8 分+ 降到后续的增量编译。

---

## 可改进的

1. **死代码删除不够谨慎**：删除 search.rs 中的"死代码"时，误删了仍在使用的 `calculate_online_relevance()`、`is_online_result_relevant()`、`contains_cjk()` 三个函数，需要通过 `git checkout` 恢复。教训：删除前应 grep 确认每个函数是否被引用。

2. **Edit 工具长文本匹配失败**：尝试用 Edit 工具删除大段代码时，`old_string` 因编码/空白差异匹配失败。最终改用 `sed` 行范围删除绕过。教训：大段删除应先用 `sed` 或直接 `Write` 重写，避免匹配问题。

3. **未及时提交阶段性成果**：阶段一的代码改动完成后没有及时 commit，导致工作区混杂了阶段二的改动和大量临时文件（tar.gz、test_*.json 等），git status 输出脏乱。

4. **settings 配置修改未提前验证**：`allowMode: "always"` 这个无效键从之前的设置中就存在，但直到用户反复反馈权限问题后才发现。教训：修改 settings 配置后应主动验证其格式是否正确。

5. **未检查专利详情页模板渲染**：权限问题修复后验证时，用了一个不存在的 patent ID "test" 测试页面，导致 58 字节的错误页面返回（"专利未找到"），未检查到模板修改的正确性。教训：测试应使用真实能渲染的路径。

---

## 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 搜索源精简方式 | 直接删除代码，保留注释标记 | 用户明确说"屏蔽"，不保留降级逻辑 |
| AI 服务商精简方式 | 删除整个 `ai_fallbacks` 结构 | 只剩 DeepSeek，无需 fallback 机制 |
| 聊天持久化方案 | 前端 `localStorage` | 无需后端改动、无需新增数据库表，对 `/ai` 和 `/patent` 这种无 session 概念的页面最轻量 |
| 重试实现方式 | 存 `retryMsg` 变量 + `retrySend()` 函数 | 比重新触发 `sendChat()` 更可控，避免输入框状态干扰 |
| 权限配置格式 | 按官方文档用 `defaultMode: "bypassPermissions"` | 唯一官方支持的跳过所有权限方式 |
