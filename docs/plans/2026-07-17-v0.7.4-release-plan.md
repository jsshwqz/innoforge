# InnoForge v0.7.4 发布计划 / Release Plan

## 目标与范围 / Goal and Scope

将 `feat/oa-fact-check` 自 `v0.7.3` 之后积累的 OA 功能、数据完整性、安全加固和可靠性修复整理为 `v0.7.4`。本计划只处理当前核心仓库，不擅自修改桌面、iOS、HarmonyOS 等独立仓库。

## 版本决策 / Version Decision

- 已发布的 `v0.7.3` 标签保持不变。
- 本批次主要完善既有 OA 工作流，没有形成新的产品阶段；根据用户确认采用 PATCH，目标版本为 `0.7.4`。
- `Cargo.toml`、`Cargo.lock`、CHANGELOG 和发布标签必须一致。

## 执行阶段 / Phases

1. 更新版本号、CHANGELOG、STATUS 和本计划的发布状态。
2. 运行 `scripts/forge_gate.ps1` 验证 Forge 工作记录。
3. 执行 `cargo fmt --all --check`、`cargo clippy --all-targets -- -D warnings`、`cargo test` 和 release 构建。
4. 对模板/静态资源执行 JS 语法、ESLint 和仓库内 Puppeteer E2E。
5. 按 `docs/release-gate-manual-e2e.md` 完成人工页面与核心输入输出验收。
6. 按 `docs/depth-quality-gate.md` 完成 4 个真实 AI 深度用例；任一失败则停止发布。
7. 将最终提交快进合入 `main`，同步 GitHub 与 Gitee，确认两个 `main` HEAD 一致。
8. 在 `main` 完整发布提交上创建并推送 `v0.7.4` 标签，等待 GitHub Release 构建完成。
9. 核对发布说明和资产；若核心仓 workflow 不能满足至少 5 个平台包，不宣布发布完成，记录缺口并补齐后复核。

## 风险与回退 / Risks and Rollback

- 禁止重写或移动既有 `v0.7.3` 标签。
- 推送前保留功能分支；若 `main` 合入后发现阻断，以新的修复提交处理，不强推、不重置远端历史。
- 标签只在全部门禁通过的最终 `main` 提交上创建；标签推送前可本地删除重建，推送后不得静默移动。
- 发布资产不足、Action 失败、双端不同步或发布后关键流程失败时，结论必须为“不可发布/发布未完成”。
