## 二、核心定位

**从"专利搜索工具"升级为"研发核查闭环系统"。**

核心目标（5 个"可"）：
- **可比对**（结构化特征卡片）
- **可解释**（最小差异点定位）
- **可追溯**（证据链系统）
- **可持续**（研发状态机，中断恢复）
- **可交付**（分层报告生成）

> 优先级最高的不是"搜得更强"，而是让产品真正专业。

## 四、已有核心功能

### 检索
- 本地 SQLite FTS5 + 在线搜索降级链（SerpAPI → Google Patents → Bing → Lens.org → 搜狗 → 本地）
- 搜索类型自动识别、国家筛选、日期范围、去重、统计、CSV 导出

### 创新验证（12 步流水线）
ParseInput → ExpandQuery → SearchWeb → SearchPatents → DiversityGate → ComputeSimilarity → RankAndFilter → DetectContradictions → ScoreNovelty → AiDeepAnalysis → AiActionPlan → Finalize

### AI 分析
- 摘要、对比、矩阵、权利要求分析、侵权预评估、批量摘要
- 多轮对话（智能上下文压缩，超 8 轮自动总结）
- 5 模型容灾 + 第一性原理/TRIZ/逆向工程等思维框架

### 平台
- Web + Android APK + MCP Server（6 工具）
- 独立仓库：patent-hub-ios / patent-hub-harmony / patent-hub-desktop

### v0.5.0 — 核心专业化（最关键版本）

**P0 必做 5 件事：**

#### 1. 技术特征卡片（Feature Card）
将专利结构化为统一格式：
- 技术问题
- 核心结构
- 关键关系
- 工艺步骤
- 应用场景

👉 支撑后续所有比对能力

#### 2. 最小差异特征定位（Minimal Diff）
- 最小差异点
- 差异类型（结构/方法/参数）
- 是否具备新颖性意义

👉 直接服务研发与专利决策

#### 3. 证据链系统（Evidence Chain）
每条结论必须附：
- 专利来源
- 权利要求编号
- 原文段落
- 置信度

👉 让结果"可被信任"

#### 4. 研发状态机（ResearchState）
```rust
struct ResearchState {
    session_id: String,
    current_hypothesis: String,    // 当前验证的创新点假设
    evidence_chain: Vec<Evidence>, // 证据链
    technical_diff: Vec<String>,   // 与现有技术的差异点
    excluded_paths: Vec<String>,   // 已排除的路径
    open_questions: Vec<String>,   // 未解决问题
    last_checkpoint: SystemTime,
}
```
支持中断恢复、多轮验证

#### 5. 分层报告生成（Report Generator）
三类输出：
- **领导版**（结论 + 建议，1 页）
- **研发版**（技术细节 + 对比矩阵）
- **专利版**（权利要求分析 + 规避建议）

格式：Markdown + PDF

## 八、Harness Engineering 方法论融入

> 借鉴 [dialectical-vibe-coding](https://github.com/whatwoods/dialectical-vibe-coding) 的核心理念：
> **Agent = Model + Harness**，Harness = Guide（前馈约束）+ Sensor（反馈修正）

### 当前已有的 Harness 机制

| 机制 | 类型 | 位置 |
|------|------|------|
| AI System Prompt（思维框架） | Guide | `routes/idea.rs` |
| Pipeline 步骤定义（13 步固定序列） | Guide | `pipeline/state.rs` |
| 代码编排 LLM（不让 LLM 做编排决策） | Guide | `pipeline/runner.rs` |
| self-review.ps1 自审查 | Sensor | `scripts/self-review.ps1` |
| cargo clippy -D warnings | Sensor | CI workflow |
| 多维推演引擎（6 维度交叉验证） | Guide + Sensor | `pipeline/steps/deep_reasoning.rs` |

### v0.5.0 需要加强的 Harness

#### 1. Steering Loop（反馈闭环）
当前多维推演是开环的（推演完就结束）。加入闭环：
```
推演结果 → 自动提取矛盾点 → 反馈下一轮推演
用户反馈 → 沉淀为规则 → 下次推演更精准
```

#### 2. Pipeline Step Sensor（步骤间验证）
每步执行后自动验证输出质量，不合格则重试或标记：
- SearchWeb 返回 0 结果 → 自动扩展查询词重试
- ComputeSimilarity 全为 0 → 标记"检索质量不足"
- AiDeepAnalysis 输出过短 → 要求重新生成

#### 3. Evidence Sensor（证据链验证）
v0.5.0 证据链系统的核心 Sensor：
- 每条结论必须附来源 → 无来源自动拒绝
- 置信度低于阈值 → 标记为"待验证"
- 引用原文与结论矛盾 → 自动预警

### 战略阶段对应

| 辩证法阶段 | 版本 | 重点 |
|-----------|------|------|
| 战略探索 | v0.1~v0.4 ✅ | 搭骨架、试模型、跑通全链路 |
| **战略相持** | **v0.5~v0.7 ← 当前** | **功能与 Harness 同步增长，强 Sensor 支撑自由度** |
| 战略收官 | v1.0 | 重型 Sensor（端到端测试、安全审计、品牌发布） |
